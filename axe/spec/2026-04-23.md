# 仕様更新 2026-04-23

## MULTI SCAN AIの役割の再定義
- **文脈スキャンの特化**: 「ここをクリック」「赤いボタン」など、自然言語や視覚的文脈の理解が必要な約18項目（`manualCheckItems` に `aiTarget: true` を付与した項目）に対してのみ直接評価を行う。
- **API送信データの最適化**: AIが得意とする項目のみを抽出し、他スキャンの結果と共にバックエンドへ送信することで、トークンの節約とAIのハルシネーション（誤判定）を防止する。
- **他スキャン結果の活用と補完**: 他の自動スキャン（BASIC / EXT / DEEP / PLAY）が検出した項目のファクトチェックを行う。
- **改善提案の生成**: 他スキャンの検出項目をインプットとし、「何が違反なのか」「どう修正すべきか」を具体的に文章化した改善提案（improvementPlan）を生成する。

## スキャン別スコア方針
- **個別スキャンの未検証扱い**: BASIC / DEEP / MULTI / PLAY / EXT の各スキャン単体では全WCAG項目を確認できないため、ツールがカバーできない、または確認不能なSCは「未検証」として扱う。
- **個別スキャン達成率の算出**: 各スキャン単体の達成率は `合格数 / (全項目数 - 未検証)` で算出する。これにより、そのスキャンエンジンが実際に判定できた項目だけを分母として達成率を可視化する。
- **TOTALスコアの算出**: TOTALスコアのみ、対象レベルの全項目数（Level A = 31項目、Level AA = 56項目）を基準の分母とする。全スキャン結果をSC単位で統合・マージし、いずれのスキャンでも判定できなかったSCは最終的に「未検証」として残る。

## AI評価処理・デバッグ仕様の更新 (2026-04-23 追記)
- **JSONパースフローの変更**: AI応答から `results` (評価結果配列) と `improvementPlan` (改善計画オブジェクト) を分離して取得するため、トップレベルのオブジェクトを抽出する `extractJsonObject()` を使用するフローとする。
- **内部デバッグ情報の状態管理**: `server.js` 内のデバッグ情報 (`_lastAiDebug` 等) は、成功応答時などの状態遷移においてオブジェクトを完全置換せず、スプレッド構文を用いて既存フィールドを保持したまま `stage: 'ai_responded'` 等のステータスを更新する。
- **SCAN_CAPABILITY_SUMMARY の定義反映**: MULTIが直接評価する対象項目（`aiTarget`）を17項目と定義し、それ以外のSCは `MULTI` 判定列を `-`（直接評価しない）とする。また、`PLAY`（キーボード・フォーカス系）や `EXT`（アクセシビリティ名・ロール等）を優先する権威スキャン（`SC_AUTHORITY`）のルールを適用する。

## UI・カード統合・レポート仕様の更新 (2026-04-23 追記)
- **初期状態のリセット**: PLAYWRIGHT および EXT スキャンは初期状態でチェック済み (`checked`) とし、クリアボタン (`clearScan()`) 押下時にもこの初期状態に復帰する。
- **スコアセルの表示形式**: スコアテーブルの項目数表示を `判定済み項目数 / 対象全項目数` （例: 45 / 56）の形式に変更し、カバレッジを明確にする。
- **エビデンスの名寄せと統合**: 同一 SC の詳細カード内において、複数のスキャンエンジン（BASIC, EXT, DEEP等）で検出された同一メッセージまたは同一箇所のエビデンスは名寄せ(`mergeEvidenceRecords`)し、複数のスキャンバッジを持つ 1つの `item-source-section` に統合して表示する。
- **AI改善案のインライン表示**: MULTI スキャンが生成した各エラーに対する個別アクション (`suggestion`) を、統合カード内のエビデンス直下に `AI改善案` として表示する。

## BATCH (一括検査) のナビゲーション一貫性 (2026-04-23 追記)
- **BATCHソースの統合**: 一括検査時のナビゲーション一貫性 (`navConsistency`) 結果を、SC 3.2.3 および SC 3.2.4 の詳細カード内に `BATCH` スキャンソースとして統合表示する。
- **権威スキャンとしてのBATCH**: SC 3.2.3 / 3.2.4 においては `BATCH` による一貫性判定を権威スキャンとし、MULTI単体の判定よりも優先してスコア・ステータスに反映する。
- **PC+SP実行時の保持**: PCとSPの両ビューで一括検査を実行した場合、SP用のナビゲーション一貫性データ (`batchMobileNavConsistency`) も独立して保持し、SP側の詳細タブおよびExcelレポート出力に正しく反映する。
- **レポート出力フォーマット**: レポート行のナビ一貫性詳細が `[object Object]` になるのを防ぐため、「対象URL」「消えたリンク」「追加リンク」などの差分内容をテキスト化して出力する。

## 追加実装・UI改善 (2026-04-24 追記)
- **手動判定の優先度とスコープ**: 手動判定（OK/NG/保留）は各個別スキャンおよびTOTALスコアにおいて、自動判定・権威スキャンよりも優先して反映される。また、判定は「PC/SP + URL」のスコープを持ち、別ビューや別URL（BATCH時）に影響を与えない。
- **EXT (IBM ACE) の判定集約**: IBM ACE の `FAIL` に加え `POTENTIAL` / `MANUAL` (要手動確認), `PASS` も収集し、`FAIL > POTENTIAL/MANUAL > PASS` の優先度で SC 別にステータスを集約する。要手動確認時は検出箇所も表示する。
- **JSONレジュームと読込ボタン**: SCAN前でもJSONを読み込めるようアクションエリアに読込ボタンを常設。`BASIC`, `DEEP`, `PLAY`, `EXT`, `MULTI` の各結果、SPスナップショット、手動判定、改善計画を復元し、UIを再描画する。
- **UI表示の改善**:
  - BATCH一括結果のURLタブ切り替え時、スコアテーブルへの自動スクロールを抑制する（初回のみスクロール）。
  - 詳細カード (`item-card`) 展開時、内容が隠れないようにリスト領域とウィンドウを自動スクロール・補正する。
- **複合SCの展開**: `1.2.x` などの複合SC（DEEP結果等）は実SC番号に展開して処理し、スコアテーブルおよび詳細タブに正しく反映する。
- **事前検出による該当なし項目の自動パス**:
  - SCAN/BATCHの実行時（事前のDOM解析時など）に、ページ内の動画・音声・入力欄の有無をチェックボックスで確認する仕様とする。BATCHのチェックボックスはページごとに設ける。
  - **要素が存在しない場合**: 以下の項目をスキャンなしで「合格（パス）」とし、レポートの備考（詳細）に「該当なし」と記載する。
    - 1.4.2 音声の制御が可能
    - 1.2.1 音声のみのコンテンツにテキスト代替がある
    - 1.2.2 動画にキャプションがある
    - 1.2.3 動画に音声解説がある
    - 3.3.1 エラーが特定されている
  - **要素を検出した場合**: 上記項目を無条件でパスとはせず、「ヒューマンチェック（要手動確認）」として扱うか、通常のスキャン（MULTI等）の対象とする。
- **予測時間の調整**: 事前に出力される予測時間（`scan-estimate`）が実際よりも長く表示される場合があるため、より正確な予測時間になるよう調整する。

## スキャン再実行とAPI安定性 (2026-04-24 追記)
- **個別再スキャンと進捗**: スコアテーブルの各スキャン行から、現在のURL・ビューに対して個別再スキャンが可能。実行中はプログレスバーで進捗（送信中/再スキャン中/反映中 等）を可視化し、多重実行を防止する。
- **API経路のフォールバック**: `/axe/` 配下での動作時など、APIがHTMLや404を返した場合は、`/api/...` と `/axe/api/...` の経路間で自動的に1回再試行し、エラーを抑制する。
- **レポート備考出力**: `not_applicable` だった項目をヒューマンチェックで「合格 (OK)」とした場合、レポート出力時の「詳細」列に自動的に「該当箇所なし」という備考を追記する（重複記載防止あり）。
