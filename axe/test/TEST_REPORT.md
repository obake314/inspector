# TEST_REPORT

スプレッドシートおよびスプレッドシートからの報告書（GAS）出力確認項目。

## T-REP-01: GASメニュー表示

- 手順
  1. 対象スプレッドシートを開く
  2. Apps Script に `axe/gas/ReportGenerator.gs` を設定
  3. シートを再読み込み
- 期待結果
  - メニュー `報告書` が表示される

## T-REP-02: 対象タブ抽出

- 手順
  1. `報告書 > アクセシビリティ評価報告書を生成` を開く
- 期待結果
  - 現行11列の結果タブが候補表示される
  - 旧7/8列の `検査項目番号` ヘッダーを持つタブも候補表示される
  - 表紙タブは候補表示されない

## T-REP-03: ダイアログ入力

- 手順
  1. `報告書 > アクセシビリティ評価報告書を生成` を開く
  2. 社名、対象サイト名、作成者、作成日、バージョンを入力
- 期待結果
  - 作成日は当日で初期入力される
  - バージョンは `1.0` が初期入力される

## T-REP-04: Docs生成と章構成

- 手順
  1. タブを選択して生成実行
- 期待結果
  - Google Docs URL が返る
  - Docsタイトルが `アクセシビリティ評価報告書 - {対象サイト名}（YYYY-MM-DD）` 形式になる
  - 以下の章が生成される
    - 表紙
    - `1. エグゼクティブサマリー`
    - `2. 評価概要`
    - `3. 問題点一覧と改善推奨`
    - `4. 改善ロードマップ`
    - `5. 推奨事項`
    - `6. 参照規格・ツール`
    - `改訂履歴`

## T-REP-05: サマリー集計

- 手順
  1. `合格` / `不合格` / `判定不能` / `未検証` / `該当なし` / `対象外` を含むシートで生成
- 期待結果
  - 総合評価結果に `合格` / `不合格` / `要確認` / `対象外` の件数が表示される
  - `判定不能` / `要手動確認` / `エラー` / `未検証` は `要確認` に集計される
  - `該当なし` / `対象外` は `対象外` に集計される

## T-REP-06: 評価概要と対象ページ表

- 手順
  1. 複数ページの現行11列シートを選択して生成
- 期待結果
  - `2.1 評価方法` に評価方法の箇条書きが表示される
  - `2.2 評価対象ページ` にページ名とURLの表が表示される
  - 複数タブを選択した場合、対象ページ表に各ページが行として表示される

## T-REP-07: 問題点一覧と改善ロードマップ

- 手順
  1. `不合格` 行を含むシートで生成
- 期待結果
  - `4. 問題点一覧と改善推奨` に不合格項目が表示される
  - `重要度` が `緊急/重大/中程度/軽微` の場合、`高/中/低` に正規化される
  - `改善案` が空欄の項目にも既定の推奨対応策が表示される
  - `5. 改善ロードマップ` に重要度別の対応内容と目標期限が表示される

## T-REP-08: 7/8/11列互換

- 手順
  1. 7列、8列、11列形式の入力シートで生成
- 期待結果
  - エラーなく読み取りできる
  - 旧7/8列形式でも SC 番号、達成基準名、適合レベル、結果、場所、検出数、詳細、改善案が可能な範囲で反映される

## T-REP-09: Drive管理API（安全確認）

- 手順
  1. `GET /axe/api/drive-cleanup`
- 期待結果
  - 使用量とファイル一覧のJSONを返す
- 注意
  - 通常テストで `deleteAll` は実行しない

## T-REP-10: Google Docs 報告書フォント

- 手順
  1. `報告書 > アクセシビリティ評価報告書を生成` から Google Docs 報告書を生成する
  2. 表紙、見出し、本文、箇条書き、各テーブルセルのフォントを確認する
- 期待結果
  - 生成された Google Docs の本文・見出し・リスト・表セルが `Noto Sans JP` で表示される
  - `ReportGenerator.gs` 内の `REPORT_DOC_FONT_FAMILY` が `Noto Sans JP` に設定されている
  - `applyDocumentFont_(body)` により、生成完了直前に全テキスト要素へフォントが再適用される
