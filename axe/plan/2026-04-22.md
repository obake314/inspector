# 実装計画 2026-04-22

## 目的

① MULTI SCAN がトークン上限・部分応答・API制限で実行不十分になった場合にスコアテーブルへ明示する。
② BASIC / DEEP / MULTI / PLAYWRIGHT / EXT の5診断を同時実行した際の集計整合性、検出説明の明確化、実行前の所要時間見積もり表示を改善する（後続計画）。
③ MULTI SCAN が自動判定できなかった原因を `APIエラー` / `モデル利用不可` / `JSON解析失敗` として明示し、実行失敗原因に `手動確認` ラベルを使わない。

## 課題

### 課題0: トークン上限でスキャンが止まったことが分からない（Fix 23）

AI API のトークン上限に達した場合、レスポンスは正常系で返るが応答が途中で切れている。現状では成功完了と区別できない。

対応:
- 各 API 呼び出し関数（`callGeminiAPI` / `callClaudeAPI` / `callOpenAIAPI`）でトークン上限フラグを検出し `{ text, tokenLimited }` を返す
- `callAI()` がフラグを伝播し `{ text, modelName, tokenLimited }` を返す
- `/api/ai-evaluate` レスポンスに `tokenLimited: true` を追加、HTTP 429 は `rateLimited: true` を追加
- クライアント: `lastMultiTokenLimited` 状態変数で管理、`clearScan()` でリセット
- 一括スキャン: `batchAITokenLimitedMap` / `batchMobileAITokenLimitedMap`（url → bool）で URL 別に管理
- スコアテーブル MULTI 行ラベルに `.score-token-warn` バッジ（amber）を表示
- MULTI SCAN ステータスメッセージにも `.ai-token-warn` バッジを表示

### 課題1: 診断結果詳細の件数が項目数と一致しない

現状:
- 5つの診断を同時に行うと、同一SCに対する不合格・未検証が複数の診断元から返る。
- 詳細タブや診断結果詳細のテーブルで、診断元ごとのカード数をそのまま数えると、項目数より多く見える。
- 「全項目数」と「緊急 + 重大 + 中程度 + 軽微 + 合格 + 該当なし + 未検証」の整合式を、SC単位の集計と検出エビデンス数のどちらで扱うかが混在している。

方針:
- 集計単位を明確に分離する。
  - `scoreSummary`: SC単位のユニーク集計。スコアテーブルと詳細タブのバッジ数に使用する。
  - `evidenceItems`: 診断元別の検出エビデンス。詳細カード内の「検出元」「検出件数」として表示するが、項目数には加算しない。
- TOTAL行と詳細タブの件数は `SC + viewport` をキーにユニーク化する。
- 同一SCに複数診断結果がある場合は、以下の優先順位で代表ステータスを決める。
  1. 不合格（重要度は `critical > serious > moderate > minor` の最重度）
  2. 未検証
  3. 合格
  4. 該当なし
- 同一SCの複数検出は、1カード内に「BASIC / DEEP / MULTI / PLAYWRIGHT / EXT のどこで検出されたか」をまとめる。
- これにより `全項目数 = 緊急 + 重大 + 中程度 + 軽微 + 合格 + 該当なし + 未検証` が詳細表示でも成立するようにする。

実装候補:
- `buildUnifiedScFindings()` を追加し、BASIC / DEEP / MULTI / PLAY / EXT の結果をSC単位に正規化する。
- `renderAllTabs()` はカード配列を直接数えず、`buildUnifiedScFindings()` のユニーク結果からタブ件数を生成する。
- 各カードには `evidenceItems` を展開し、診断元別の詳細を表示する。

### 課題2: EXT SCAN / PLAYWRIGHT SCAN の検出内容と検出箇所が不明瞭

現状:
- EXT SCAN と PLAYWRIGHT SCAN は、検査関数や外部ツール由来のメッセージがそのまま表示される箇所があり、何を検出したのか、どの箇所を見ればよいのかが分かりにくい。
- 特に EXT は `IBM_ACE` / `EXT_NATIVE` / `EXT_CDP` の検出元が混在し、説明粒度にばらつきがある。

方針:
- PLAYWRIGHT / EXT の結果を表示前に共通スキーマへ正規化する。
  - `scanSource`: `PLAYWRIGHT`, `IBM_ACE`, `EXT_NATIVE`, `EXT_CDP`
  - `detectedContent`: 何を検出したか
  - `detectedLocation`: どこで検出したか
  - `reason`: なぜ問題または未検証なのか
  - `evidence`: セレクタ、HTMLスニペット、イベント種別、対象属性など
  - `suggestion`: 確認・修正の観点
- 詳細カードでは以下の見出しを固定表示する。
  - 検出内容
  - 検出箇所
  - 判定理由
  - 検出ツール
  - 修正・確認観点
- PLAYWRIGHT の各検査関数は、SC番号だけでなく「検査対象」「確認したDOM/操作」「失敗条件」を返す。
- EXT の各検出元は、以下の説明ルールに揃える。
  - `IBM_ACE`: IBM Equal Access の ruleId、message、target、HTMLスニペットを明記
  - `EXT_NATIVE`: 独自DOM検査の検査条件、該当セレクタ、検出値を明記
  - `EXT_CDP`: Chrome DevTools Protocol で検出したイベント種別、対象要素、キーボード操作への影響を明記

実装候補:
- `normalizePlayFinding(result)` / `normalizeExtFinding(result)` を追加する。
- サーバー側で不足している情報は `server.js` の PLAY / EXT 結果生成時に追加する。
- クライアント側の `renderCardItem()` に、正規化済みの「検出内容」「検出箇所」を優先表示する分岐を追加する。

### 課題3: SCAN / BATCH 実行前に概算の予測時間を表示したい

現状:
- DEEP / MULTI / PLAYWRIGHT / EXT のチェック状態、URL件数、PC/SP選択によって所要時間が大きく変わる。
- 実行前にユーザーが総時間を見積もれないため、特にBATCHやPC+SPで待ち時間の見通しが立てづらい。

方針:
- SCAN / BATCH ボタン近くに所要時間の概算を表示する。
- 表示はURL入力、BATCH URL行数、PC/SPビュー、各スキャンチェックボックスの変更に追従して更新する。
- BASIC は常に実行される前提で見積もりに含める。
- PC+SP は viewport 数を2として計算する。
- BATCH は有効URL件数をページ数として計算する。
- 表示例:
  - `予測時間: 約3〜7分 / 1ページ / PC / BASIC + DEEP + EXT`
  - `予測時間: 約24〜48分 / 4ページ / PC+SP / BASIC + MULTI + PLAYWRIGHT`
- URL未入力時は `URL入力後に予測時間を表示` とする。

初期見積もり値:

| スキャン | 1ページ・1ビューあたりの目安 |
|---|---:|
| BASIC | 10〜30秒 |
| DEEP | 2〜4分 |
| MULTI | 1〜2分 |
| PLAYWRIGHT | 3〜5分 |
| EXT | 2〜4分 |

実装候補:
- `getSelectedScanTypes()` を追加し、単一/一括のチェック状態を共通取得する。
- `estimateScanDuration({ urlCount, viewportMode, scanTypes })` を追加する。
- `renderScanEstimate()` を追加し、`#singleScanEstimate` / `#batchScanEstimate` へ描画する。
- 対象イベント:
  - URL入力
  - BATCH textarea入力
  - PC/SP radio変更
  - DEEP / MULTI / PLAYWRIGHT / EXT checkbox変更
  - 単一/一括モード切替
- MULTI がAIキー未設定でdisabledの場合は、見積もり対象から除外し、その旨を補足表示する。

### Fix 26: MULTI API制限/部分応答と詳細タブ集計の追加方針

問題:
- MULTI が HTTP 429 / quota exceeded で実行できていない場合も `manual_required` に丸められ、エラーとして見えにくい。
- AI応答が一部しか返っていない場合、未返答分が黙って手動確認になり、スキャン品質の低下が見えない。
- MULTI の説明が `reason` だけで、検出内容・根拠・セレクタが弱い。
- `resultTabs-pc` はMULTI未実行時にもMULTI項目を未検証カードとして描画する経路があり、スコア表TOTALと見た目がずれる。

方針:
- `/api/ai-evaluate` はAPI制限/認証/接続エラーを `success:false` として返す。
- クライアントはAI APIエラーを `manual_required` フォールバックカードとして保持しつつ、MULTI行に `API制限` / `APIクォータ不足` / `APIエラー` を表示する。
- `partialResults` / `missingCount` / `warning` を追加し、部分応答は `部分応答` として表示する。
- AIプロンプトに `reason` / `evidence` / `selector` / `suggestion` を要求し、詳細カードに表示する。
- `normalizeScoreToTarget()` を共通化し、スコア表TOTALと詳細タブバッジを同じSC単位正規化で算出する。
- 設定パネルには「APIキー単体では残トークン数を取得できない」旨を明記する。

### Fix 27: `.tab-num` と詳細カードがBASIC表示に戻る問題の追加方針

問題:
- `renderAllTabs()` の `items.map()` で UNIFIED カードに必要な `sc` / `entry` を受け取っておらず、DEEP/PLAY/EXT の統合カード描画時に例外が発生しうる。
- SCAN中に BASIC 完了直後の `displayResults()` が詳細タブを即時描画するため、後続スキャン完了前のBASIC表示が残りやすい。

方針:
- `items.map(({ src, type, data, dkey, sc, level, entry }, idx) => ...)` として、UNIFIEDカード描画へ必要データを渡す。
- `displayResults(results, url, { renderTabs: false })` を追加し、SCAN/BATCH中の中間BASIC結果では詳細タブを描画しない。
- 全スキャン完了後の `renderAllTabs()` だけで、`.tab-num` と詳細カードを TOTAL 統合結果に更新する。

### Fix 29: MULTI SCAN 実行失敗原因の明示方針

問題:
- MULTI SCAN が API エラーやAI応答解析失敗で自動判定できなかった場合、UI 上のバッジが `手動確認` となり、何が失敗したのか判断できない。
- GPT-5 など選択モデルが利用不可の場合も、一般的な手動フォールバック表示に見えてしまう。

方針:
- `/api/ai-evaluate` の API 呼び出し失敗は `aiErrorType: api_error` または `model_unavailable` として返す。
- AI応答をJSONとして解析できなかった場合は `aiErrorType: json_parse_failed` として返し、応答先頭を `responsePreview` に残す。
- クライアントの MULTI 行バッジは `APIエラー` / `モデル利用不可` / `JSON解析失敗` を表示する。項目単位の `manual_required` は残すが、実行失敗原因の表示に `手動確認` は使わない。

### Fix 30: 詳細カード開閉クリック範囲の調整方針

問題:
- `.item-card` 全体に開閉ハンドラがあるため、展開後に `.item-source-section` 内の検出内容や検出箇所をクリックするとカードが閉じてしまう。

方針:
- 開閉ハンドラを `.item-card` から `.item-header` に移す。
- `.item-source-section` / `.item-locations` は閲覧・テキスト選択用領域として扱い、クリックしても開閉状態を変えない。

### Fix 31: DEEP SCAN 2.5.8 スクリーンリーダー専用要素除外方針

問題:
- DEEP SCAN の SC 2.5.8 ターゲットサイズ検査で、`sr-only` などスクリーンリーダー専用の視覚的に隠した要素が 24×24px 未満として検出される。

方針:
- `check_2_5_8_target_size()` で class/id/data 属性にスクリーンリーダー専用マーカーを持つ要素とその配下を除外する。
- 除外対象は `sr-only` / `screen-reader` / `screen-reader-text` / `visually-hidden` / `assistive-text` 等に限定し、通常表示される小さいリンク・ボタンは引き続き検出する。
- 違反表示には class/id を含め、除外されない要素の追跡性を上げる。

### Fix 32: GPT APIエラー原因追跡の方針

問題:
- GPT APIエラー時に `APIエラー` とだけ表示され、キー不正、モデル利用不可、クォータ、課金、非対応パラメータ、トークン制限のどれか判別しにくい。
- GPT-5 を `max_tokens` で呼ぶと、モデル側で非対応パラメータとして失敗する可能性がある。

方針:
- GPT-5 / o3 / o1系は `max_completion_tokens` を使う。
- OpenAI API呼び出しに `X-Client-Request-Id` を付与し、失敗時は OpenAI の `requestId` と合わせてレスポンスに返す。
- `/api/ai-evaluate` のエラーレスポンスへ `causeHint`、`status`、`code`、`errorType`、`param`、`requestId`、`clientRequestId`、`model` を含める。
- UIのエラー詳細文に原因候補を含め、OpenAI Dashboard / Billing / Rate limits / Project権限 / モデル利用可否のどこを確認すべきか分かるようにする。

### Fix 33: WordPress標準lightbox keyframesの点滅候補除外方針

問題:
- DEEP SCAN の SC 2.3.1 点滅検査で、WordPress標準実装の `turn-on-visibility` / `turn-off-visibility` / `lightbox-zoom-in` / `lightbox-zoom-out` が「点滅を含む可能性のあるアニメーション」として検出される。

方針:
- `check_2_3_1_three_flashes()` で上記4件の keyframes 名を allowlist として除外する。
- allowlist外の `opacity: 0` / `visibility: hidden` を含む keyframes は引き続き要確認として検出する。

### Fix 34: DEEP/PLAY 誤検知抑制の追加方針

問題:
- SC 1.4.12 テキスト間隔検査で、`.screen-reader-text` などスクリーンリーダー専用要素のクリップが違反として出る。
- SC 2.4.11 フォーカス隠れ検査で、通常時は隠れていてfocus時に表示される設計の要素まで誤検知する余地がある。
- SC 2.3.1 点滅検査で、現在ページ上で実際には使われていない `internSlideFade` などのCSS keyframes が検出される。

方針:
- `check_1_4_12_text_spacing()` にスクリーンリーダー専用マーカー除外を追加する。
- `check_2_4_11_12_focus_obscured()` / `pw_check_2_4_11_focus_obscured()` は Tab 後の focus 状態を待ってから判定し、focus時に表示される要素は違反にしない。focus時にも非表示・0×0・clip状態の要素は違反にする。
- `check_2_3_1_three_flashes()` は、現在ページの要素に実際に適用されており `animation-duration > 0` の keyframes だけを点滅候補として扱う。
