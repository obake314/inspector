### Fix 23: MULTI SCAN トークン上限をスコアテーブルに明示

**問題**: AI API のトークン上限に達してスキャンが途中終了した場合、レスポンスが途切れていても UI 上では正常完了と区別できなかった。
**対応**: 各 AI プロバイダーでトークン上限を検出し、スコアテーブルの MULTI 行とステータスメッセージに警告バッジを表示する。

#### server.js

- `callGeminiAPI()`: `response.candidates?.[0]?.finishReason === 'MAX_TOKENS'` を検出し `{ text, tokenLimited }` を返すよう変更
- `callClaudeAPI()`: `message.stop_reason === 'max_tokens'` を検出し `{ text, tokenLimited }` を返すよう変更
- `callOpenAIAPI()`: `completion.choices[0].finish_reason === 'length'` を検出し `{ text, tokenLimited }` を返すよう変更
- `callAI()`: `tokenLimited` フラグを各プロバイダー関数から受け取り `{ text, modelName, tokenLimited }` として返すよう変更
- `/api/ai-evaluate`: `aiTokenLimited` 変数を追加し上限検出時はサーバーログに警告出力。レスポンスに `tokenLimited: true` を追加
- `/api/ai-evaluate`: API 呼び出し時の HTTP 429 エラーを `isRateLimit` として検出し、フォールバックレスポンスに `rateLimited: true` を追加

#### public/index.html

- `let lastMultiTokenLimited = false` 状態変数を追加
- `clearScan()` で `lastMultiTokenLimited = false` にリセット
- 単一スキャン: `runAIEvaluation()` のレスポンスから `!!data.tokenLimited` を `lastMultiTokenLimited` に格納。`!silent` 時のステータス文字列に `⚠ トークン上限` バッジ（`.ai-token-warn`）を追加
- 一括スキャン: `aiTokenLimitedMap`（`url → bool`）をバッチループ内で構築。`runBatchViewport()` の戻り値に `aiTokenLimitedMap` を追加
- PC 一括: `batchAITokenLimitedMap`、SP 一括: `batchMobileAITokenLimitedMap` として格納
- `clearScan()` で両マップをリセット
- 一括詳細表示 `showBatchDetail()`: `lastMultiTokenLimited = !!batchAITokenLimitedMap[r.url]` を設定してから `renderScanScoreTable()` を呼ぶ。SP 側も同様
- 単一・一括すべての `renderScanScoreTable()` 呼び出しに `multiTokenLimited: lastMultiTokenLimited`（または `snapshot.multiTokenLimited`）を `options` として追加
- `renderScanScoreTable()`: `const multiTokenLimited = !!options.multiTokenLimited` を読み取り、MULTI ラベルのサブテキスト内に `<span class="score-token-warn">⚠ トークン上限</span>` を条件付きで挿入
- `snapshot.multiTokenLimited` を `runViewportPipeline()` 内で MULTI 完了後に保存し、後続の PLAY / EXT フェーズでも継承

#### public/css/style.css

- `.score-token-warn`: スコアテーブル MULTI ラベル内用の amber バッジ（`background: #d97706`、9px、ラウンド角）
- `.ai-token-warn`: MULTI SCAN ステータスメッセージ内用の amber バッジ（11px）

### 課題1: SC単位ユニーク集計（詳細タブ件数整合）

**問題**: DEEP / PLAY / EXT が同一 SC を複数ソースから返すと、タブバッジが SC 数より多くなっていた。
**対応**: `buildScUnifiedMap()` で DEEP/PLAY/EXT 結果を SC をキーに統合し、`renderAllTabs()` で UNIFIED type として1件扱いにする。バッジは `badgeMap[key] ?? 0`（computeTotalScore と同一の SC 単位集計値）を使用。

#### public/index.html

- `buildScUnifiedMap(deep, play, ext, scSet)` 追加: DEEP/PLAY/EXT 結果を SC→エントリマップに統合。ステータス優先順: fail > unverified > pass > na
- `renderUnifiedCard(no, sc, level, entry, applyDecisionFn)` 追加: SC 名タイトル・ソース別エビデンスセクションを含む統合カードを生成
- `renderAllTabs()` の DEEP/PLAY/EXT バケット充填を `scUnifiedMap.forEach()` に置き換え
- カード描画ロジックの `type === 'deep'` / `type === 'play'` / `type === 'ext'` 分岐を削除し `type === 'unified'` 分岐を追加
- タブバッジ行を `items.length > 0 ? items.length : badgeMap[key]` → `badgeMap[key] ?? 0` に変更

### 課題3: SCAN / BATCH 実行前の予測時間表示

- `estimateScanDuration({ urlCount, viewportMode, scanTypes })` 追加
- `renderScanEstimate()` 追加: 単一・一括それぞれの URL 数・ビューポート・チェック状態から予測時間を算出して表示
  - URL 未入力時は "URLを入力すると予測時間を表示します" を表示
  - MULTI が disabled の場合は MULTI を除外して計算
- `bindScanEstimateListeners()` 追加: `urlInput` / `batchUrls` input、viewport radio change、scan checkbox change をバインド
- モード切替ハンドラ (`modeButtons.forEach`) から `renderScanEstimate()` を呼ぶよう追加
- スクリプト末尾で `bindScanEstimateListeners()` / `renderScanEstimate()` を初期実行

#### public/index.html (HTML)

- `#singleScanEstimate` div: `#singleScanControls` 内に配置済み
- `#batchScanEstimate` div: `#batchScanControls` 内に配置済み

#### public/css/style.css

- `.scan-estimate`: 12px、navy-400 色のインライン表示スタイル追加済み

### Fix 26: MULTI API制限/部分応答の明示と詳細タブ集計整合

**問題**:
- MULTI が HTTP 429 / quota exceeded で実行できていない場合も `manual_required` に丸められ、エラーとして見えにくかった。
- AI応答が一部しか返らない場合、未返答分が黙って手動確認になり、スキャン品質の低下が見えなかった。
- MULTI 詳細カードの表示が `reason` 中心で、検出内容・根拠・セレクタが弱かった。
- `resultTabs-pc` はMULTI未実行時にもMULTI項目を未検証カードとして描画する経路があり、スコア表TOTALと見た目がずれる場合があった。

**対応**:
- `server.js`
  - `AI_MAX_OUTPUT_TOKENS = 8192` を追加し、Gemini / Claude / OpenAI の出力トークン上限を拡張
  - `classifyAIError()` / `buildAIErrorResponse()` を追加し、API制限・クォータ不足・認証エラーを分類
  - `/api/ai-evaluate` は API 制限/認証/接続エラー時に `success:false` と HTTP `429` / `401` / `502` を返すよう変更
  - AIプロンプトで `reason` / `evidence` / `selector` / `suggestion` を要求し、曖昧な説明を避ける指示を追加
  - AI応答の未取得項目を `partialResults` / `missingCount` / `warning` として返すよう変更
- `public/index.html`
  - `lastMultiIssue` / `batchAIIssueMap` / `batchMobileAIIssueMap` を追加
  - AI APIエラー時はクライアント側で `manual_required` フォールバック結果を生成し、`API制限` / `APIクォータ不足` / `APIエラー` バッジを表示
  - MULTI 詳細カードに `selector` / `evidence` / `reason` / `suggestion` を「セレクタ」「根拠」「理由」「改善案」として表示
  - `normalizeScoreToTarget()` を追加し、スコア表TOTALと詳細タブバッジを同じSC単位正規化で算出
  - MULTI未実行時はMULTI項目を詳細タブへ自動描画しないよう変更
  - 設定モーダルに「APIキー単体では残トークン数を取得できない」旨を追記
- `public/css/style.css`
  - `.ai-status-note` を追加し、MULTI API警告の詳細文を表示

### Fix 27: `.tab-num` と詳細カードがBASIC表示に戻る問題を修正

**問題**:
- `resultTabs-pc` の `.tab-num` が TOTAL スコアではなく BASIC 直後の値に見える状態が残っていた。
- 詳細タブに BASIC 以外の DEEP / PLAY / EXT 結果が出ず、BASIC のみ表示されていた。

**原因**:
- `renderAllTabs()` の `items.map()` で UNIFIED カード描画に必要な `sc` / `level` / `entry` を受け取っておらず、DEEP/PLAY/EXT 統合カードの描画時に例外が発生しうる状態だった。
- SCAN / BATCH の BASIC 完了直後に `displayResults()` が詳細タブを即時描画しており、全スキャン完了前のBASIC表示が残りやすかった。

**対応**:
- `items.map(({ src, type, data, dkey, sc, level, entry }, idx) => ...)` に変更し、UNIFIEDカードへ `sc` / `level` / `entry` を渡すよう修正
- `displayResults(results, url, options)` に `renderTabs: false` を追加
- SCAN/BATCH の中間BASIC結果では詳細タブを描画せず、全スキャン完了後の `renderAllTabs()` で TOTAL 統合結果を描画するよう変更

### Fix 29: MULTI SCAN 実行失敗原因の表示を明確化

**問題**:
- MULTI SCAN が自動判定できなかった場合、UI バッジが `手動確認` と表示され、APIエラー・モデル利用不可・JSON解析失敗のどれか分からなかった。

**対応**:
- `server.js`

  - AI応答をJSONとして解析できない場合は `aiErrorType: json_parse_failed` / `detailLabel: JSON解析失敗` / `responsePreview` を返すよう変更
- `public/index.html`
  - MULTI 行の実行失敗バッジを `APIエラー` / `モデル利用不可` / `JSON解析失敗` に統一
  - 実行失敗原因のバッジとして `手動確認` を表示しないよう変更
